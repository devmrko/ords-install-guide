# 04. HA — 2노드 + OCI Load Balancer + 사설 TLS

## 4.1 아키텍처

```
       Internet (443/80)
              │
              ▼
     ┌──────────────────┐
     │  OCI Load Balancer  │  ← 사설 TLS 인증서로 종단
     │   (Flexible shape) │     HTTP(80) → HTTPS(443) 리다이렉트
     └──────┬─────┬─────┘
            │     │  HTTP 8080 (VCN 내부)
       ┌────▼─┐ ┌─▼────┐
       │ ORDS │ │ ORDS │   ← 둘 다 standalone + systemd
       │  N1  │ │  N2  │
       └───┬──┘ └──┬───┘
           │       │
           ▼       ▼
       ┌──────────────┐
       │     ADB      │  (myadb_high)
       └──────────────┘
```

- ORDS 는 stateless → sticky session 불필요
- LB healthcheck: `GET /ords/_/landing` → **HTTP 302** (인증 redirect; ORDS 살아있다는 신호)
- TLS 는 LB 에서만 종단, backend 는 평문 HTTP 8080 (VCN 내부)

## 4.2 사전 (네트워크 + 호스트 방화벽)

- VCN / Public Subnet (LB 용) / Private Subnet (ORDS 노드 용) 분리 권장
- Security List:
  - **LB → ORDS**: ingress 8080 from LB subnet CIDR
  - **인터넷 → LB**: ingress 443 / 80 from 0.0.0.0/0
  - **ORDS → ADB**: egress 1522 to ADB private endpoint (또는 NAT)
- **호스트 firewalld** (Oracle Linux / RHEL 기본 ON): 노드별로 8080 개방 필요.
  `00_prereq.sh` 가 자동으로 처리하지만, 수동 확인:
  ```bash
  sudo firewall-cmd --add-port=8080/tcp --permanent && sudo firewall-cmd --reload
  ```
  안 열면 LB backend health 가 영원히 `CONNECT_FAILED` (CRITICAL).

## 4.4 LB 구성 — 트랙 A: 웹 콘솔 (수동)

### 4.4.1 사설 인증서 등록 (LB 에 직접 attach — 권장)

콘솔에서 LB 생성 도중 cert 를 같이 등록할 수 있음.

> **참고**: OCI Certificate Service 에 별도 등록도 가능하지만, **provider v8.x 의
> `oci_certificates_management_certificate` 는 IMPORTED PEM 입력을 지원하지 않음**
> (CSR/Internal CA 발급만 가능). 콘솔 Cert Service 에 import 한 cert 의 OCID 를
> LB listener 가 참조하는 형태는 가능하지만, 가장 단순한 경로는 **LB 자체 cert 리소스에 PEM 직접 attach**.

### 4.4.2 Load Balancer 생성

1. **Networking → Load Balancers → Create Load Balancer**
2. Type: **Public** (또는 Private), Shape: **Flexible** (10–100 Mbps)
3. **Backends**: Add Backends → ORDS 노드 IP × 2, port `8080`
4. **Health Check**:
   - Protocol: HTTP
   - Port: `8080`
   - URL path: `/ords/_/landing`
   - **Status code: 302** (ORDS 가 인증 redirect 로 응답하므로 200 아니라 302)
5. **Listener**:
   - Protocol: **HTTPS**, Port `443`
   - **Certificate**: "Load Balancer Managed Certificate" 선택 후 cert/private key/chain PEM 업로드
     (또는 미리 OCI Cert Service 에 등록해두고 OCID 참조)
6. **Create**
7. (선택) HTTP 80 → HTTPS 301 리다이렉트 ruleset 추가

5분 후:
```bash
curl -k https://<LB_PUBLIC_IP>/ords/_/landing
```

## 4.5 LB 구성 — 트랙 B: Terraform (권장)

`terraform/main.tf` 가 위 단계를 전부 코드화. 인증서도 같이 import 함.

### 4.5.1 준비
```bash
# 컨트롤 머신에서 (ORDS 노드 아니어도 됨, OCI CLI만 있으면 됨)
oci setup config            # ~/.oci/config 생성
terraform version           # 1.3+
```

### 4.5.2 사설 인증서 PEM + region 설정
`.env` 에:
```bash
# region: ~/.oci/config DEFAULT 와 다른 region 으로 배포할 때 필요
OCI_REGION=ap-seoul-1                   # 비우면 ~/.oci/config DEFAULT 사용

# 두 가지 cert 출처 중 택1
# (A) PEM 으로 새로 LB 에 attach
TLS_CERT_PEM=/etc/ords/tls/cert.pem
TLS_KEY_PEM=/etc/ords/tls/privkey.pem
TLS_CHAIN_PEM=/etc/ords/tls/chain.pem   # 없으면 빈 값
OCI_CERT_NAME=ords-private-cert
OCI_CERT_OCID=                          # 비워두면 (A) 모드

# (B) 이미 OCI Cert Service 에 등록된 cert 의 OCID 를 listener 에 연결만
# OCI_CERT_OCID=ocid1.certificate.oc1..xxxx   # 채우면 (B) 모드, PEM 무시
```

### 4.5.3 실행
```bash
./run.sh ha-tf plan
./run.sh ha-tf apply
./run.sh ha-tf output
# lb_public_ip = [ "129.x.x.x" ]
# cert_ref     = "ords-private-cert"   (또는 cert OCID)
```

### 4.5.4 동작 검증
```bash
LB_IP=$(./run.sh ha-tf output -json | jq -r '.lb_public_ip.value[0]')
curl -k https://$LB_IP/ords/_/landing   # HTTP 302 정상
```

`-k` 없이 cert 신뢰까지 끝까지 확인하려면 hostname 으로 접속해야 함 → §4.6 참고.

### 4.5.5 인증서 회전(rotation)
- (A) 모드: `.env` 의 PEM 경로 갱신 후 `./run.sh ha-tf apply`
  → `oci_load_balancer_certificate` 가 `create_before_destroy` 라 무중단 교체
- (B) 모드: OCI Cert Service 에서 cert version 회전 → LB 가 최신 version 자동 사용 (terraform apply 불필요)

### 4.5.6 정리
```bash
./run.sh ha-tf destroy
```
LB-attached cert 는 LB 와 함께 삭제됨 (별도 보호 lifecycle 없음).

### 4.5.7 ⚠️ State 파일 보안

local backend(기본)는 인증서/키를 `terraform.tfstate` 에 평문 저장. 운영 사용 전 반드시:
```bash
cd terraform
cp backend.tf.example backend.tf
vi backend.tf          # OCI Object Storage 버킷/region 채움

```

## 4.6 클라이언트가 hostname 으로 접속하게 만들기

LB 는 IP 만 줍니다. 사설 cert 의 SAN(`ords.test.local` 같은) 과 IP 가 안 맞으면
브라우저/curl 이 TLS 에서 막힙니다 (`no alternative certificate subject name matches target host name`).
두 가지 방법:

### 4.6.1 PoC / 내부 테스트 — `/etc/hosts`
접속할 머신 몇 대만 알면 충분할 때.

```bash
# 클라이언트 머신에서:
echo "<LB_IP>  ords.test.local" | sudo tee -a /etc/hosts

curl https://ords.test.local/ords/_/landing   # cert SAN 과 일치 → TLS OK
```

curl 만이라면 `--resolve` 옵션으로 한 번씩 우회도 가능:
```bash
curl --resolve ords.test.local:443:<LB_IP> https://ords.test.local/ords/_/landing
```

**한계**: 접속할 머신마다 직접 등록해야 함. JVM/Docker 컨테이너 내부에서는 무시되는 경우 있음.

### 4.6.2 운영 — 진짜 DNS A 레코드

공인/사내 DNS 에 A 레코드 등록.

| 도구 | 등록 방법 |
|---|---|
| OCI DNS Zone | Networking → DNS Management → Zones → 해당 zone → Add Record → A record → `ords.test.local`, IPv4 = LB IP, TTL 300 |
| Cloudflare/Route53 | 콘솔에서 A 레코드 추가, value = LB IP |
| 사내 BIND | zone 파일에 `ords.test.local. 300 IN A <LB_IP>` 추가 후 reload |

등록 후:
```bash
dig +short ords.test.local      # LB IP 떠야 함
curl https://ords.test.local/ords/_/landing
```

> 운영에서 cert SAN 은 **실제 사용할 hostname** 으로 발급하세요 (예: `ords.example.com`).
> `ords.test.local` 같은 `.local` 은 mDNS 와 충돌 가능 + 공인 CA 발급 불가.

### 4.6.3 LB IP 가 바뀔 수 있나
- OCI LB 는 IP 가 고정 (재생성하지 않는 한). 즉 destroy + apply 안 하면 안 바뀜
- 그래도 DNS TTL 은 300~600s 정도로 두면 향후 마이그레이션 용이

## 4.7 운영 팁

- **롤링 재시작**: LB 에서 한 노드 drain (backend offline) → `systemctl restart ords` → 응답 정상 확인 → 반대편 동일
- **세션 한도**: `JDBC_MAX_LIMIT × 노드수 ≤ ADB sessions`
- **로그 집계**: 노드별 `journalctl -u ords` 와 `/etc/ords/config/logs/` → 중앙 로그(예: OCI Logging) 로 보내는 것 권장
- **APEX**: 이 가이드 범위 밖. 같은 ORDS 인스턴스에 추가 가능하지만 별도 챕터로 분리 추천
